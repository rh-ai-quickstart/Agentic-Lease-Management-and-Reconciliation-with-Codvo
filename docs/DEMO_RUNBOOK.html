NeIO LeasingOps on OpenShift

Live deploy on Red Hat partner lab cluster 264c5202-codvoqs. Captured 2026-05-15.

This runbook walks the partner team through deploying NeIO LeasingOps end-to-end on OpenShift with GPU-served Granite 3.3 2B via vLLM and LlamaStack. Every command below was executed on this cluster, and every block of "expected output" is the real output that came back.

Infrastructure GREEN vLLM on GPU GREEN LlamaStack → Granite chat GREEN Frontend reachable Upload accepted Worker poll-loop has an intermittent stall after first agent. See findings.

4Deploy Postgres and Redis

The chart's default ships Bitnami subcharts that don't play well with OpenShift restricted-v2 SCC. Plain manifests are more reliable. Service names match the chart's defaults so no extra wiring is needed.

Manifest (drop into scripts/leasingops-data-services.yaml) 
---
apiVersion: v1
kind: Service
metadata:
  name: neio-leasingops-postgresql
  namespace: leasingops
spec:
  selector: { app: postgresql }
  ports:
    - name: pg
      port: 5432
      targetPort: 5432
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: postgresql
  namespace: leasingops
spec:
  replicas: 1
  selector: { matchLabels: { app: postgresql } }
  template:
    metadata: { labels: { app: postgresql } }
    spec:
      containers:
        - name: postgres
          image: docker.io/library/postgres:15-alpine
          ports: [{ containerPort: 5432 }]
          env:
            - name: POSTGRES_USER
              valueFrom: { secretKeyRef: { name: neio-leasingops-secrets, key: POSTGRES_USER } }
            - name: POSTGRES_PASSWORD
              valueFrom: { secretKeyRef: { name: neio-leasingops-secrets, key: POSTGRES_PASSWORD } }
            - name: POSTGRES_DB
              value: leasingops
            - name: PGDATA
              value: /var/lib/postgresql/data/pgdata
          volumeMounts: [{ name: data, mountPath: /var/lib/postgresql/data }]
      volumes: [{ name: data, emptyDir: {} }]
---
apiVersion: v1
kind: Service
metadata:
  name: leasingops-redis
  namespace: leasingops
spec:
  selector: { app: redis }
  ports: [{ name: redis, port: 6379, targetPort: 6379 }]
---
apiVersion: apps/v1
kind: Deployment

  name: redis
  namespace: leasingops
spec:
  replicas: 1
  selector: { matchLabels: { app: redis } }
  template:
    metadata: { labels: { app: redis } }
    spec:
      containers:
        - name: redis
          image: docker.io/library/redis:7-alpine
          ports: [{ containerPort: 6379 }]
          command: ["sh", "-c"]
          args: ['exec redis-server --requirepass "$REDIS_PASSWORD" --save "" --appendonly no']
          env:
            - name: REDIS_PASSWORD

oc apply -f scripts/leasingops-data-services.yaml
oc rollout status deploy/postgresql -n leasingops --timeout=180s
oc rollout status deploy/redis -n leasingops --timeout=120s
Captured output
service/neio-leasingops-postgresql created
deployment.apps/postgresql created
service/leasingops-redis created
deployment.apps/redis created
deployment "postgresql" successfully rolled out
deployment "redis" successfully rolled out
Don't name the Redis Service redis exactly. Kubernetes auto-injects Service env vars like REDIS_PORT=tcp://... for every Service in the namespace. The chart constructs REDIS_URL=redis://...@$(REDIS_HOST):$(REDIS_PORT) and the substitution picks up the auto-injected REDIS_PORT instead of the ConfigMap value. Result: ValueError: Port could not be cast to integer value as 'tcp:' in the worker. Using leasingops-redis avoids the collision.

6Deploy LlamaStack with the remote vLLM provider

helm install llamastack rh-ai-quickstart/llama-stack \
  --version 0.7.3 \
  --namespace leasingops \
  --set pgvector.enabled=false \
  --set models.remote-llm.enabled=true \
  --set models.remote-llm.url=http://granite-3-3-2b-instruct-vllm:80/v1

oc rollout status deploy/llamastack -n leasingops --timeout=180s
Captured output
NAME: llamastack
LAST DEPLOYED: Fri May 15 00:42:00 2026
NAMESPACE: leasingops
STATUS: deployed
deployment "llamastack" successfully rolled out
Three traps in one step:
  1. pgvector.enabled=false is mandatory. The chart's default tries to bring up a pgvector PVC the application doesn't use and the install fails.
  2. The key is models.remote-llm (hyphen), and it must be set with both enabled=true and url=.... Setting only vllm.url (as some older docs suggest) is a no-op.
  3. The URL must end in /v1. LlamaStack's remote::vllm provider appends path segments directly to base_url; without the /v1 suffix the upstream returns 404 Not Found on chat completions.

7Register the Granite model in LlamaStack

LlamaStack does not auto-discover models from vllm.url. Models must be registered explicitly via POST.

oc exec -n leasingops deploy/llamastack -- python3 -c "
import urllib.request, json
body = json.dumps({
  'model_id': 'ibm-granite/granite-3.3-2b-instruct',
  'provider_id': 'remote-llm',
  'provider_model_id': 'ibm-granite/granite-3.3-2b-instruct',
  'model_type': 'llm'
}).encode()
req = urllib.request.Request(
  'http://localhost:8321/v1/models',
  data=body, method='POST',
  headers={'Content-Type': 'application/json'})
r = urllib.request.urlopen(req)
print('OK', r.status, r.read().decode())

Findings / gotchas from this real run

#FindingEffectFix in this runbook
1Chart uses containerSecurityContext in values-openshift.yaml but the deployment templates read .Values.{api,worker,app}.securityContextSecurity context was silently dropped, PodSecurity admission rejected pods, Helm reported "deployed" with no workload (the "nothing seemed to be deployed" symptom)Renamed the field in the chart; committed.
2Chart hard-codes fsGroup/runAsUser/runAsGroup = 1000OpenShift restricted-v2 SCC rejects (UID must be in namespace range 1000790000+)Override file nulls out runAsUser/runAsGroup; grants anyuid SCC to the SA so the image's appuser (uid 1000) is allowed.
3Container seccompProfile: RuntimeDefault is converted by an admission webhook into a forbidden deprecated annotationPod creation rejectedOverride file sets securityContext: { seccompProfile: null, ... }.
4vLLM Service port is 80, not 8080LlamaStack→vLLM 404s if you use the old portUse http://granite-3-3-2b-instruct-vllm:80/v1.
5LlamaStack remote::vllm provider expects base_url to include /v1Without it: Error code: 404 - {'detail': 'Not Found'}Append /v1 on the install URL.
6LlamaStack does not auto-register models from vllm.urlCalling /v1/chat/completions with a model name returns "Model not found"One-time POST to /v1/models.
7Chart's llama-stack install was failing on missing pgvector PVCInstall fails--set pgvector.enabled=false.
8Naming the Redis Service redis collides with Kubernetes auto-injected service env vars (REDIS_PORT=tcp://...)Worker constructs malformed REDIS_URL, dies with Port could not be cast to integer valueRenamed Service to leasingops-redis; set cache.host=leasingops-redis in overrides.
9Chart template defined URL env vars BEFORE the source vars they referenceKubernetes $(VAR) only resolves vars defined earlier in the env list; URLs contained literal $(POSTGRES_PORT)Reordered env vars in api/deployment.yaml and worker/deployment.yaml so URLs come AFTER source vars. Committed to chart.
10Chart references SA neio-leasingops but doesn't create itReplicaSet stuck on serviceaccount not foundoc create sa neio-leasingops before install.
11Pull secret value is configured at chart-level imagePullSecrets, not global.imagePullSecretsAPI/worker pods stuck in ImagePullBackOff--set 'imagePullSecrets[0].name=acr-pull-secret' at top level.
12App probes default to /api/health; Next.js app doesn't serve that pathContainer is killed by liveness probe every minute → CrashLoopBackOffOverride sets app.livenessProbe.path: / and app.readinessProbe.path: /.
13Uploads PVC is RWOCan't have api and worker on different nodes both mounting; second replica stuck ContainerCreatingOverride sets api.replicas: 1 and app.replicas: 1.
14Worker poll loop sometimes stops emitting after processing the first agent in a chain (matches Bala's earlier "pipeline chain bug" note)Document is upload→contract_intake but then idles in pending:term_extractionNot a fix yet. Workaround: oc rollout restart deploy/neio-leasingops-worker -n leasingops drains a stuck queue. Real fix needs a worker code patch; file an issue on the worker image to log every poll cycle and surface the stuck state.
